# 청정 시간 (Green Hour) — 구현 계획서

> 기반 문서 : `청정시간_심층조사.md` (조사·근거)
> 작성일 : 2026.05.11
> 대상 : 2026 기후부 AX 아이디어 경진대회 데모 + 시범 서비스 MVP

---

## 0. 솔직한 범위 평가

심층조사에서 그린 풀 비전을 그대로 만들 수는 없다. **단독 구현 가능 영역**과 **외부 협력 필수 영역**을 먼저 가른다.

### 0-1. 단독 구현 가능 (외부 협력 없이도 작동)

| 기능 | 근거 |
|---|---|
| **EPSIS 발전 믹스 실시간/이력 조회** | 전력거래소 Open API 공개. API 키 신청만으로 사용 |
| **시간대별 탄소강도 계산** | EPSIS 발전 믹스 × 발전원별 배출계수(공시 자료) → 자체 산출 |
| **24시간 청정 시간 예측** | 과거 EPSIS + 기상청 일사·풍속 예보로 시계열 모델 학습 |
| **청정 시간 알림 UI** | Next.js 웹 + PWA Push (또는 React Native) |
| **시뮬레이션 절감 효과 표시** | 자체 추정 로직 |
| **NILM 데모** | 공개 데이터셋(REDD, UK-DALE) + 사전학습 NILMFormer |
| **데모용 가상 가정 데이터** | 시뮬레이션 LP 데이터 생성 |

### 0-2. 외부 협력 필수 (MVP에서는 mock·시뮬레이션)

| 기능 | 차단 요인 | MVP 대응 |
|---|---|---|
| 한전 마이데이터 실제 LP | 한전 사업자 협약 필요 | 시뮬레이션 LP 데이터 생성, 또는 사용자 직접 업로드 |
| SmartThings/ThinQ 가전 제어 | 개발자 등록·가전 페어링 | 데모는 mock 시뮬레이션 |
| 한전 캐시백 자동 신청 | 한전 협약 | 화면만 mock |
| 국민DR 자동 정산 | 전력거래소 협약 | 화면만 mock |
| 탄소중립포인트 자동 적립 | 환경부 cpoint.or.kr 협약 | 화면만 mock |
| NILM 한국 가정 미세조정 | 한국 가정 라벨링 데이터 부재 | 영국·미국 공개 데이터로 사전학습 결과 시연 |

### 0-3. 결론

**경진대회 데모 = Tier 1(실제 동작) + Tier 2(데모 시연)**.
실서비스 진입은 한전·환경부와의 협약 단계가 별도 필요.

## 1. 빌드 우선순위 (3단계)

### Tier 1 — 진짜 동작하는 핵심 (필수, 4주)

> 심사위원 앞에서 실제 데이터로 작동해야 하는 기능.

1. **실시간 탄소강도 대시보드** — EPSIS 데이터 → 시간대별 gCO₂/kWh 그래프
2. **24시간 청정 시간 TOP 3 예측** — 다음날 가장 깨끗한 시간 산출
3. **가전별 시간 추천 시뮬레이터** — 사용자가 세탁기/식기세척기/EV 선택 + 사용 시간 창 입력 → 최적 시간 + 절감 효과 즉시 표시
4. **회피 알림 로직** — LNG 피크 시간 자동 감지

### Tier 2 — 데모용 시연 기능 (선택, +3주)

> mock·시뮬레이션 기반이지만 사용자 경험을 입증하는 기능.

5. **푸시 알림** — PWA Web Push 또는 FCM
6. **시뮬레이션 가정 등록** (페르소나 3종 — 맞벌이/EV/바우처)
7. **NILM 가전 분리 시연** (공개 데이터셋 기반)
8. **주간 리포트 화면** — "이번 주 절감 = 소나무 2.1그루"
9. **4개 제도 자동 통합 화면** (mock UI)

### Tier 3 — 정책 제안 자료 (발표용)

10. **국가중점데이터 4종 통합 흐름도** (다이어그램)
11. **시뮬레이션 정책 효과 계산기** — "사용자 N명 참여 시 LNG 회피 규모 X MW"

## 2. 기술 스택 결정

| 계층 | 기술 | 근거 |
|---|---|---|
| **프론트** | Next.js 14 (App Router) + TypeScript + Tailwind + shadcn/ui | 데모 + PWA + SSR 한 코드베이스 |
| **차트** | Recharts (간단) 또는 Visx (고급) | 시간대별 탄소강도 그래프가 핵심 |
| **푸시** | PWA Web Push API (1차) → Firebase Cloud Messaging (2차) | 네이티브 앱 없이도 푸시 가능 |
| **백엔드** | Python FastAPI + APScheduler | EPSIS 주기적 fetch + ML 추론 서빙 |
| **DB** | PostgreSQL + TimescaleDB (시계열) | EPSIS 시계열 누적 |
| **ML** | PyTorch + ONNX Runtime / NILMFormer 공개 가중치 | 사전학습 모델 추론만 (학습은 Tier 2+) |
| **시계열 예측** | 단순 LSTM 또는 Prophet (1차) → Transformer (확장) | 1차는 baseline 모델로 충분 |
| **인프라** | Docker Compose (로컬 데모) → AWS Seoul / NCP (배포) | 데모는 로컬도 가능 |
| **모노레포** | pnpm workspaces 또는 Turborepo | apps/web · apps/api · packages/shared |

> **모바일 네이티브 앱 — Tier 1에서는 만들지 않음.** PWA로 모바일 시연 충분. 필요해지면 React Native(Expo)로 확장.

## 4. 단계별 마일스톤 (10주)

### Week 0 — 셋업 (4일)

- [ ] 모노레포 부트스트랩 (pnpm workspaces)
- [ ] Next.js + FastAPI 스켈레톤
- [ ] PostgreSQL + TimescaleDB Docker 셋업
- [ ] EPSIS Open API 키 발급 ([epsis.kpx.or.kr](https://epsis.kpx.or.kr))
- [ ] 기상청 Open API 키 발급
- [ ] 발전원별 배출계수 JSON 정리 (석탄·LNG·원자력·태양광·풍력·기타)
- [ ] CI 셋업 (lint·typecheck·test)

**검증** : EPSIS 발전 믹스 1회 호출해서 응답 받기

### Week 1~2 — 데이터 파이프라인 (Tier 1)

- [ ] EPSIS API 클라이언트 (실시간 발전 믹스 + 과거 이력)
- [ ] APScheduler로 5분마다 EPSIS fetch → TimescaleDB 저장
- [ ] 발전원 → 탄소강도 환산 로직 (carbon-calc 패키지)
- [ ] 기상청 동네예보 클라이언트
- [ ] 데이터 검증 노트북 (Jupyter) — 1주일치 탄소강도 계산 결과 확인

**검증** : 실제 어제 데이터로 "어제 청정 시간 TOP 3" 산출이 직관과 맞는지

### Week 5~6 — 웹 대시보드 (Tier 1)

## 7. 위험 + 대응

| 위험 | 가능성 | 대응 |
|---|---|---|
| **EPSIS API 응답 지연·장애** | 중 | 5분 cron으로 자체 캐시. API 死 시 마지막 데이터 보여주기 + 정적 백업 |
| **EPSIS API 키 발급 지연** | 중 | Week 0에 즉시 신청. 대체로 Electricity Maps API 키 병행 신청 |
| **24h 예측 정확도 부족** | 중 | 1차 baseline(Prophet) 일단 동작 → 정확도 부족 시 LightGBM·LSTM 단계적 개선. 데모는 정확도보다 **개념 입증** |
| **NILM 데모 시간 부족** | 중 | Tier 2이므로 후순위. 안 되면 사전학습 모델 결과만 정적 시각화 |
| **PWA 푸시 iOS 한계** | 낮 | iOS 16.4+ 만 지원. 데모는 Android 휴대폰 사용 |
| **팀원 시간 부족** | 높 | Tier 1만 해도 충분. Tier 2·3은 여유분 |
| **발표 당일 인터넷 끊김** | 낮 | 영상 백업 + 로컬 데이터로 작동하는 빌드 별도 보관 |
